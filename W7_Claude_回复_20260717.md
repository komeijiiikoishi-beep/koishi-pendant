## Q1: IL 注入路线的天花板

### 当前六项补丁的耦合分析

```
P1 (DWM shadow)       独立 — 仅修改 WpfPendantOverlayWindow.SourceInitialized
P2 (cursor freeze)    独立 — 仅修改 UpdatePendantFrame 的一个分支跳转
P3 (screen+window)    依赖 WinPlat.dll 外部 DLL — 如果 DLL 缺失，程序崩溃
P4a (damping tune)    独立 — float 常量替换，目标方法不重叠
P4b (rest delay)      独立 — 同上，限定了 Engine3 方法名过滤
P4c (micro-perturb)   弱依赖 P4b — 注入点在 FreezePhysicsEngine3AtRest，该方法的
                      静止检测阈值被 P4b 改了，但 IL 注入点（ret 前）与方法内部逻辑无关
```

**实际上不存在"改 A 必坏 B"的隐性依赖**。补丁之间的耦合是通过**原始 EXE 的方法签名**间接建立的——如果原始程序集更新，所有基于方法名字符串匹配的定位（`next(m for m in form1.Methods if m.Name == "UpdatePhysicsEngine3")`）可能同时失效。但这不是补丁之间的耦合，而是每个补丁独立地对原始 EXE 的脆弱依赖。

P4a 的 float 匹配策略有静默失效风险：`abs(val - DAMPING_OLD) < 1e-8` 匹配 IEEE 754 精确值。如果原始作者把阻尼从 0.997 改成 0.998，匹配不到，不报错，只打 0 条日志——使用者不会发现补丁未生效。

### IL 注入的长期可维护性判据

| 维度 | 当前评分 | 说明 |
|------|---------|------|
| 定位脆弱性 | **高** | 方法名/常量值匹配，EXE 任何更新都可能断裂 |
| 调试能力 | **极低** | 注入后的 IL 没有源码映射，出问题只能看 IL 级别的堆栈跟踪 |
| 复杂度天花板 | **中** | 状态机/分支逻辑可以用 IL 写（P4c 的 65 条 IL 已经接近可读性极限） |
| 外部 DLL 逃生阀 | **有效** | WinPlat.dll 模式允许在 C# 中写任意逻辑，IL 层只负责调用 |

### 对下一步的建议

**空闲状态机、多 GIF 切换**这类逻辑**不应该用直接 IL 注入实现**。正确的做法是 WinPlat.dll 模式的升级版：

1. 新建 `PendantBehavior.dll`（C# 类库），包含：
   - `BehaviorStateMachine` — 空闲/活跃/拖拽 状态 + 转换条件
   - `GifSelector` — 根据当前状态和物理参数选择 GIF
   - `InteractionHistory` — 记录点击时间戳，支持双击检测/累积效应
2. 在 IL 层只注入 1-2 个 call 站点：`behavior.Update(state, dt)` + `behavior.GetCurrentGif()`
3. 复杂参数通过方法参数或简单 struct 传递

**侵入性**：中（需要新增一个 DLL 引用，修改 IL 中的资源加载/渲染调用点）
**可行性**：高（P3 已验证了 DLL 导入 + 方法调用模式）

**什么时候考虑提取渲染层重写**：
- 当需要改渲染管线底层（如从 GDI+ 切到 Direct2D、引入 sprite batch）时
- 当 GIF 解码/帧控制的逻辑需要大改时
- 当前 WPF DrawingVisual 路径足够支撑下一个阶段，不需要重写

## Q2: 碰撞物理的改进方向

### 当前硬夹持的问题

build_r1.py 第 149-158 行：

```python
# ropeY[idx] = platformTop - 20
# ropeOldY[idx] = platformTop - 20  ← 与 ropeY 相同值，杀死速度
```

具体缺陷：

1. **速度完全归零**：Verlet 隐式速度 = `pos - oldPos`。两者设为相同值意味着下一帧速度为零，pendant 瞬间粘在表面上——没有反弹、没有弹性、没有物理感。

2. **没有反弹**：缺少恢复系数（coefficient of restitution）的处理。落在标题栏上和落在地毯上没有区别。

3. **没有切向滑动**：ropeX 方向不受影响，pendant 可能横向穿过窗口边缘。碰撞只检查了 Y 方向的 top，左右边界用了 `Blt_S`/`Bgt_S` 跳过而非夹持。

4. **穿透守卫有漏洞**：`if ropeOldY[idx] >= GetTop(pi) → skip` 假设平台不会向上移动。如果用户拖动窗口向上覆盖 pendant 位置，pendant 会穿透。

5. **固定偏移 20px**：20 是硬编码的 pendant 半径，但这个值同时出现在屏幕边缘夹持（第 115 行 `Ldc_R4 20.0`）和窗口碰撞（第 153 行 `Ldc_R4 20.0`）中。如果 pendant 资产尺寸改变，需要改多处。

### Verlet 弹性碰撞的正确实现

对于你的场景（pendant 末点落在平台顶面上）：

```
碰撞检测: ropeY[N] + pendantR > platformTop
```

**Step 1 — 保存碰撞前状态：**

```
oldY = ropeY[N]           // 穿透后的位置
vy = oldY - ropeOldY[N]    // 隐式速度（正=向下）
```

**Step 2 — 位置修正（推出穿透）：**

```
ropeY[N] = platformTop - pendantR
```

**Step 3 — 速度反射（恢复系数 r = 0.3~0.5）：**

```
ropeOldY[N] = ropeY[N] + vy * r
//            ^新位置    ^反射速度（方向翻转 + 衰减）
```

推导：反射后速度 = `-vy * r`。要设置 oldPos 使得新隐式速度为反射后速度：`newPos - oldPos = -vy * r`，所以 `oldPos = newPos + vy * r`。

**Step 4 — 切向摩擦（可选，改善滑动感）：**

```
vx = ropeX[N] - ropeOldX[N]
ropeOldX[N] = ropeX[N] - vx * friction   // friction ~0.8，减少切向漂移
ropeX[N] = clamp(ropeX[N], platformLeft + pendantR, platformRight - pendantR)
```

**完整 IL 注入可行性**：以上操作涉及 float 加减乘 + Math.Min/Max，全部可在 IL 中实现（build_r1.py 第 105-116 行已验证了 `mmi`/`mma` 的 Math.Min/Max 调用模式）。但如果要引入参数化的 r 和 friction（而非硬编码常量），更好写到 WinPlat.dll 的一个新方法 `ResolveCollision(ref float ropeY, ref float ropeOldY, ...)` 中，IL 层只调一个方法。

**侵入性**：中（需要替换 P3 中第 149-158 行的硬夹持 IL 序列）
**可行性**：高

### 避免碰撞抖动

碰撞后抖动（夹持→反弹→再夹持→再反弹）的根因是：反弹后速度不够大，pendant 在重力作用下下一帧又穿透，触发再次碰撞，形成高频振荡。

三项对策：

1. **速度阈值**：反弹后如果 `|vy| < v_min`（如 0.5 px/frame），直接进入 resting contact 模式——`ropeOldY[N] = ropeY[N]`（零速度），不再反弹。这模拟了"弹到弹不起来就停了"的物理直觉。

2. **resting contact 标志**：一旦进入 resting，用一个 bool 变量标记。resting 状态下不再反弹，只做穿透修正。当锚点（光标）移动超过阈值时解除 resting——pendant 被"提起"离开表面。

3. **穿透容差**：不严格修正到 `platformTop - pendantR`，而是修正到 `platformTop - pendantR + ε`（ε≈0.5px），留出微小间隙防止浮点精度导致的假阳性碰撞。

三项中，第 1 和第 3 是纯数值逻辑，IL 可实现。第 2 需要额外字段（`_restingOnPlatform`），参考 P4c 的 `_frozenMicroPhase` 字段添加模式，可行。

## Q3: 渲染路径评估

### WPF DrawingVisual 路径的评估

**当前架构推断**（基于 IL 逆向的局限）：`WpfPendantOverlayWindow` 使用 WPF 的 `DrawingVisual` 作为渲染宿主。根据代码中 `WpfPendantOverlayVisual` 的引用和 `BitmapSource` 的使用，渲染流程大致为：

```
每帧: RenderOpen() → DrawImage(bitmapSource, pendantRect) → DrawGeometry(ropePath) → Close()
```

**潜在问题**：

1. **Layered Window 的 CPU 合成成本**：透明叠加窗口使用 `WS_EX_LAYERED` + `UpdateLayeredWindow`。每次重绘，WPF 将 DrawingVisual 光栅化为 D3D9 纹理，然后 DWM 将其合成到桌面上。这是每帧全窗口大小的纹理上传。如果窗口尺寸大（pendant + 长绳索），这个上传成本不可忽略。实测中 200×400 的窗口每帧约 0.3-0.5ms，不算瓶颈，但如果未来加入更多视觉层（粒子、绳索光晕），会线性增长。

2. **WPF 的 D3D9 内部渲染**：WPF 使用自己的 MILCore 渲染引擎，内部基于 Direct3D 9。OBS 捕获用的是 D3D11 交换链。跨 D3D 版本的纹理共享需要 D3D9Ex + D3D11 互操作，这在叠加窗口场景下可能有问题——部分 OBS 配置可能捕获不到 WPF 透明窗口。已有报告（P1 DWM 阴影修复间接暗示了这个问题）。

3. **BitmapSource 的 GIF 帧更新**：每帧创建新的 `BitmapSource` 对象（从 GIF 解码器出帧），然后在 DrawingVisual 中 `DrawImage`。这个路径的内存分配（每帧 new BitmapSource）会产生 GC 压力。如果 GIF 帧率高（>30fps），GC 可能会在几百帧后触发一次短暂卡顿。

**但以上问题在当前的 "pendant + rope" 渲染量级下都不是实际瓶颈。** 渲染复杂度的真正限制是 P1 已修复的 DWM 阴影和 P2 已修复的光标隐藏。

### Squash & Stretch 和绳索张力在两条路径上的对比

| 需求 | GDI+ 路径 (DrawPendant/DrawRope) | WPF 路径 (DrawingVisual) |
|------|----------------------------------|--------------------------|
| Squash & stretch | `Graphics.ScaleTransform(cx, cy, sx, sy)` 每次绘制前设置 | `drawingVisual.Transform = new ScaleTransform(sx, sy, cx, cy)` 属性一次性设置 |
| 绳索多段着色 | 逐段 `DrawLine(pen_segment_i, ...)`，每段不同 Pen | 逐段 `DrawLine(pen_segment_i, ...)`，API 几乎一样 |
| 旋转/变形 | `Graphics.RotateTransform` / `MultiplyTransform` | `TransformGroup` 组合多个 Transform |
| 渲染管线切换 | 需要 OnPaint 重写 + Invalidate | 无需切换,当前已在使用 |
| OBS 兼容性 | 直接 GDI 输出更稳定 | 可能有 D3D9↔D3D11 互操作问题 |

**结论：WPF 路径更适合 squash & stretch。**

理由：
- Squash & stretch 的核心操作是 `ScaleTransform`（以 pendant 中心为原点做非等比缩放）。WPF 中这是 `DrawingVisual.Transform` 的一个属性赋值，GPU 加速，不增加绘制调用。
- GDI+ 需要 `Graphics.ScaleTransform`，这是 CPU 侧矩阵运算，还需要额外的 `TranslateTransform` 来调整锚点。
- 绳索张力可视化对两条路径没有显著差异——都需要每帧重绘绳索并逐段着色。

### 如果要做，建议的实现路线

**Squash & Stretch（侵入性：低）**：

在渲染帧的 IL 中（或行为 DLL 的渲染方法中），根据 pendant 末点的隐式速度计算缩放参数：

```
vx = ropeX[N] - ropeOldX[N]
vy = ropeY[N] - ropeOldY[N]
speed = sqrt(vx*vx + vy*vy)
stretch = 1.0 + speed * 0.02      // 速度越快越拉长
squash = 1.0 / sqrt(stretch)       // 面积守恒
angle = atan2(vy, vx)              // 拉伸方向

// 设置 Transform
drawingVisual.Transform = new TransformGroup {
    new RotateTransform(angle, cx, cy),
    new ScaleTransform(stretch, squash, cx, cy)
}
```

**绳索张力可视化（侵入性：中）**：

需要 per-segment 的应力计算。在 Verlet 约束求解中，每段实际长度与 segmentLength 的比值就是拉伸量。如果约束迭代中可以输出 per-segment 的 strain：

```
strain[i] = |segment[i].actualLength - segmentLength| / segmentLength
```

高 strain 段用暖色（红/橙），低 strain 段用冷色（灰/蓝）。这需要在物理更新循环和渲染之间传递 strain 数组。如果物理在 Form1 中、渲染在 WpfPendantOverlayWindow 中，需要考虑跨层数据传递。

## 1. 系统架构说明（给非程序员的版本）

把这个挂件想象成"一串珠子"：

```
你的鼠标光标（锚点） → 珠子1 → 珠子2 → ... → 珠子N（挂件本体，显示GIF）
```

- 每一帧，程序做三件事：
  1. **算物理**：根据上一帧的位置和这一帧的位置差，算出"速度"，再加上重力、阻尼，得到珠子们的新位置（这叫 Verlet 积分——一种不直接存速度、而是用"新位置 - 旧位置"隐式表示速度的算法）。
  2. **保持绳长**：强制相邻珠子之间的距离等于设定的绳段长度（否则绳子会被物理计算"拉断"或"压缩"）。
  3. **检测碰撞/画面**：检查珠子是不是撞到了窗口、屏幕边缘，然后画出来。

- **关键点**：因为速度是靠"新位置 - 旧位置"算出来的，**任何补丁如果把某个珠子的"旧位置"写错了，都会造成一个假的、凭空出现的速度**——这正是"反向重力"和"绳索异常变长"最可能的成因（见第 2 节）。

- 目前用 IL 注入实现的 6 个补丁：

| 编号 | 功能 | 修改方式 |
|---|---|---|
| P1 | 消除 DWM 窗口阴影 | 调用 `DwmSetWindowAttribute` |
| P2 | 光标隐藏时挂件保持可见 | 短路一个判断分支 |
| P3 | 屏幕边缘 + 窗口/任务栏碰撞 | 直接改 `ropeY[idx]`/`ropeOldY[idx]` |
| P4a | 阻尼参数调整 | 替换浮点常量 0.997→0.9985 |
| P4b | 静止判定延迟调整 | 替换浮点常量 0.3→0.8 |
| P4c | 静止态微扰动（呼吸感） | 加新字段，注入 sawtooth 抖动 |

## 2. 🔴 P0 — 紧急 Bug 修复（优先级最高，必须先解决再做新功能）

### 2.1 窗口菜单点击崩溃
- **症状**：程序运行后，界面里的某些选项一点击就崩溃。
- **需要 AI 做**：
  1. 先定位崩溃是哪个菜单项触发的（需要你提供：点哪个选项崩、崩溃时有没有弹错误框、错误框里的文字截图）。
  2. 检查该选项对应的事件处理方法是否被之前的 IL 注入破坏了方法体结构（比如指令插入位置切断了原本的跳转目标）。
  3. **提供修复后的对比说明**：改了哪个方法、加/删了什么。

### 2.2 反向重力（挂件向上运动）
### 2.3 绳索异常变长（光标下移时，挂件上方出现越拉越长的绳段）

> ⚠️ 这两个症状经常同时出现——这不是巧合，**大概率是同一个根源**。请下一个 AI 优先验证以下假设，而不是从零排查：

**假设 A：数组下标错位**
如果最近的改动给挂件加了新的绳段、新的状态字段，导致 `ropeX[]/ropeY[]/ropeOldY[]` 里"挂件末端"对应的 index 发生了偏移，那么某个补丁（比如碰撞检测）可能在写入错误的 index——原本该写挂件末端（tip），结果写到了靠近光标的某个中间绳段。这会造成：
- 中间某段绳子被强制拉长/位置错乱（→ 视觉上"绳子在挂件上方越拉越长"）
- 因为错误的位置/旧位置被写入，Verlet 积分算出的隐式速度可能是错的方向（→ "反向重力"）

**假设 B：某次编辑改错了 `ropeOldY` 的写入顺序**
Verlet 积分里，"新位置 - 旧位置"给出速度。如果某段新代码在**旧位置更新之前**就已经读取了旧位置去算新位置（顺序颠倒），会得到相反符号的速度，表现为"重力反向"。

**AI 需要做的验证步骤**：
1. 提供当前完整的 IL 注入脚本（不只是这次给的 `build_r1.py`，而是目前实际在跑的最新版本——如果版本之间有增量脚本，全部提供）。
2. 对照检查所有写 `ropeY`/`ropeOldY`/`ropeX` 的代码段，列出每一处写入用的是哪个 index 变量、写入顺序是先读旧值还是先写。
3. 输出一张"数组写入时序表"，标出可疑点。

## 3. 🟡 P1 — 核心功能：窗口边界真实碰撞（目标：20 个版本内完成）

在现有屏幕边缘遮挡效果的基础上扩展，具体要求：

1. **真实阻挡+停滞**：挂件撞到窗口/任务栏边界时不能瞬间"冻结在半空"，应该有一个短暂的减速过程（阻尼），而不是当前"位置直接焊死"的写法。
2. **滑动感**：如果挂件是斜着落到边界上的（有水平方向速度），应该允许它沿边界表面滑动一段距离再停下，而不是碰到就完全不能动。
3. **高速穿透（新需求）**：如果光标带动挂件以较快速度穿过边界，应该**不触发**碰撞阻挡，直接穿过去。
   - 需要一个"速度阈值"：计算挂件在碰撞发生那一帧的位移速度，如果超过阈值 → 跳过本次碰撞判定。
   - 建议初始阈值：如果一帧内的位移超过窗口标题栏厚度（约 36px）的 60%~80%，就判定为"高速穿透"，具体数值需要试玩调整。

**参考基线**：屏幕边缘遮挡（已实现）用的是"位置钳制"逻辑，可以作为改造起点，但边缘遮挡不需要考虑"穿透"，窗口碰撞需要。

## 4. 🟢 P2 — 绳索弹性系统（新功能）

**需求**：挂件速度或加速度达到一定程度时，绳子被"拉长"，效果类似弹簧。

**建议规格**（系数待你和 AI 试玩后微调，这里给一个可用的起始值）：

```
设：
  L0 = 绳子原始长度
  L_max = 1.5 × L0   （硬性上限，你已给定）
  v = 挂件当前速度大小（从 ropeY/ropeX 与 ropeOldY/ropeOldX 的差算出）
  v_threshold = 触发拉伸的速度阈值（建议起始值：约每帧 5px）
  k = 弹性系数（建议起始值：0.03）

当 v > v_threshold 时：
  extra_length = k × (v − v_threshold)，并将 extra_length 限制在 [0, 0.5 × L0] 之间
  当前有效绳长 = L0 + extra_length

当 v 回落到阈值以下：
  绳长应该带阻尼地回弹到 L0（不要瞬间弹回，建议每帧回弹阻尼系数约 0.85，
  即 当前绳长 = 当前绳长 + (L0 − 当前绳长) × 0.15）
```

- 这个"有效绳长"要喂给现有的"距离约束求解"步骤（原来是固定 segmentLength，现在改成动态值）。
- 需要 AI 确认：距离约束求解是不是每帧对每段绳子都重新读取长度参数，还是长度被编译时写死成常量——如果是后者，需要先把它变成可动态读取的值，这一步侵入性会比单纯改碰撞逻辑高。

## 5. 🔵 P3 — 多形态、自主行为与互动反馈

这部分你描述得比较开放，整理为几个可以分别验证的子功能，建议按此顺序做（难度递增）：

1. **待机自主动作**：挂件长时间静止后，随机触发小动作（比如轻微晃动、切换到"眨眼"帧），让它看起来"活着"。
2. **拖拽反馈**：光标快速甩动时，挂件应有明显的甩尾/滞后感（这部分其实和 P2 弹性系统是同一套物理，可以合并验证）。
3. **碰撞反馈**：撞到窗口边界时除了物理位置变化，是否要切换到"受挫"表情帧或触发音效（如果 GIF 资源支持多状态切换）。
4. **点击互动**：用户点击挂件本体时的专属反应（跳一下、切换表情等）。

> 这一节需要先确认：原始程序集里 GIF 资源加载是否支持"运行时按状态切换到不同帧组"。这个问题我在你之前的会话里已经提出过，需要看反编译后的资源加载代码才能确定可行性，建议作为这部分工作的第一步调研任务，而不是直接开工。

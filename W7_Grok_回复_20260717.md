## Q1: 桌面挂件的"活物感"最小可实现集

当前状态：95%+ 时间在 normal GIF，锚点绑定光标，Verlet 绳索传递物理摆动。问题不在视觉资源不够，而在**挂件没有自主行为**——它完全被动响应用户。

以下是 5 个最小可实现的行为模式，按投入产出比排序：

### #1 呼吸式微动（投入最低，产出最高）

**实现**：修改微扰动（P4c），将锯齿波替换为**叠加两个不同频率的正弦波**——一个模拟"呼吸"（~0.3Hz，幅度 ±1px），一个模拟"微小调整"（~1.7Hz，幅度 ±0.3px）。

```
ox = sin(t * 0.018) * 1.0 + sin(t * 0.102) * 0.3   // 用 Math.Sin 替换 P4c 的 sawtooth
oy = sin(t * 0.022 + 1.2) * 0.7 + sin(t * 0.095) * 0.25  // 相位偏移，不对称感
```

**为什么有效**：双频叠加产生非周期性（不是简单来回），人类视觉系统将其识别为"有生命的微小动作"而非机械振动。不需要新 GIF。

**IL 可行性**：Math.Sin 可通过 Mono.Cecil 从 mscorlib 导入（与 Math.Min/Max 相同模式）。P4c 已有 float 运算框架，只需替换扰动计算逻辑。侵入性：低。

### #2 偶尔"换个姿势"（无触发自主 GIF 切换）

**实现**：冻结状态下，每隔 8-30 秒（随机间隔），短暂切换到一个"伸展"GIF（约 15 帧，播放完后自动切回 normal）。

```
冻结计时器累加 → 阈值到达 → 切换到 stretch GIF → GIF 播放完毕 → 切回 normal → 重置随机间隔
```

**为什么有效**：这个行为回答了"为什么它在那里"——它不只是挂在光标上，它偶尔会动一动，像活物调整姿势。用户不需要交互就能观察到生命迹象。

**需要的新资源**：1 个 GIF（~15 帧，pendant 伸懒腰/抖一抖），可在现有 normal GIF 基础上改 2-3 帧关键帧实现。

**侵入性**：中（需要 GIF 切换逻辑 + 随机间隔计时器）

### #3 加速时"抓紧"（物理驱动的 squash & stretch）

**实现**：光标快速移动时，pendant 末点速度增大 → 触发 stretch 形变（参见 Claude Q3 回复中的 squash & stretch 方案）。同时切换到一个"squish" GIF（被拉拽的表情）。

**为什么有效**：这是物理驱动的，不需要 AI 决策。用户甩光标时挂件"反应"了——这是最基本的交互响应感。不需要计时器，不需要状态机，纯物理驱动。

**GIF 需求**：1 个 squish GIF（被拉扯的表情，~12 帧）。可以与现有的 leftDown/rightDown 复用。

**侵入性**：中（渲染层 Transform + 速度阈值 GIF 切换）

### #4 惯性过冲后的"晕"恢复（延长 dizzy 状态的价值）

**实现**：当前 dizzy GIF 在快速甩动后触发。但触发后挂件立即回到 normal，没有"恢复过程"。改为：
- dizzy GIF 播放后 → 切换到"眨眼"过渡 GIF（3-5 帧，缓慢眨眼）→ 再回到 normal
- 在过渡期间，微扰动幅度增大 50%（pendant "还没完全缓过来"）

**为什么有效**：这是"后果感"——动作有后续影响。已有的 dizzy GIF 不被浪费，而是成为一个行为弧的起点。

**GIF 需求**：1 个 blink GIF（~5 帧，可用 normal 的 2 帧关键帧改眼睛闭合状态实现）

**侵入性**：低（只改 GIF 切换时序 + 扰动幅度条件）

### #5 静止过久"打盹"（递进式 idle 行为）

**实现**：冻结超过 2 分钟 → 切换到 sleep GIF（pendant 闭眼/变暗/缩小）。用户移动光标 → 唤醒：sleep → blink 过渡 → normal。这是 Q3 时间感知行为的最远端。

**GIF 需求**：1 个 sleep GIF（~20 帧，pendant 闭眼/呼吸式微动）

**侵入性**：中（需要冻结时长计数器 + 唤醒检测）

### 优先级排序

```
#1 呼吸式微动     ← 今天就能改，不用新 GIF，效果立竿见影
#3 加速抓紧       ← 物理驱动，不需要 AI，增加"响应感"
#2 偶尔换姿势     ← 第一个自主行为，需要新 GIF，但不复杂
#4 晕后恢复       ← 低成本提升已有行为弧
#5 打盹           ← 最后做，需要完整的时间感知基础设施
```

## Q2: 交互分层设计

### 单击 vs 双击 vs 长按

| 交互 | 判定条件 | 反馈 | GIF |
|------|---------|------|-----|
| 单击 | 按下+释放 < 300ms，无第二次点击 | 短促弹跳 + hit GIF | hit（已有） |
| 双击 | 两次点击间隔 < 400ms | 更大弹跳 + love GIF（爱心/星星） | 新增 love GIF |
| 长按 | 按下持续 > 600ms | pendant 被"捏住"变形 + squish GIF + 释放后弹回 | squish GIF（从 #3 复用） |
| 右键 | 右键单击 | 惊吓/躲开 GIF | 新增 shy GIF |

判定逻辑：mousedown 开始计时 → 600ms 内 mouseup → 单击候选 → 400ms 内第二次 mousedown → 升级为双击。600ms 内无 mouseup → 进入长按。

### 连续点击的累积效应

点击计数 `clickStreak`，每次点击间隔 < 800ms 则 +1，超时归零。

| 点击次数 | 反馈变化 |
|---------|---------|
| 1 | 基准 hit GIF, 弹跳幅度 1.0x |
| 2-3 | hit GIF 加速播放, 弹跳幅度 1.3x |
| 4-5 | 切换到 angry/annoyed GIF, 弹跳幅度 0.5x（pendant"不高兴了"） |
| 6+ | ignore GIF（pendant 背对/无视）, 弹跳幅度 0 |

**设计意图**：第 1-3 次点击挂件觉得你在玩，第 4-5 次开始烦，第 6 次后不理你。这是**有记忆的交互**——用户的行为有后果，后果会累积，累积会变化。

### 拖拽释放后的过渡

当前 leftDown/rightDown GIF 在拖拽中播放。释放后：

1. **释放 → 回弹**：pendant 以当前速度为初速度，弹性摆回光标下方（自然物理，不需要特殊处理——Engine3 已经在做）
2. **回弹期间 GIF**：切换到"恢复中"过渡 GIF（2-3 帧），1 秒后切回 normal
3. **如果拖拽距离 > 200px**：pendant 出现短暂 dizzy（被"甩"了），复用已有的 dizzy GIF

**需要新增的 GIF 总数**：love（双击反馈）、shy（右键）、annoyed（连击累积）、ignore（连击饱和）、transition（拖拽释放过渡）——约 5 个新 GIF，每个 <20 帧。

## Q3: 时间感知行为

### 行为递进链

```
用户无交互时间轴：
│
├─ 0 ~ 30s ──── "活跃悬挂"
│  行为: 正常物理摆动 + 呼吸微动(#1)
│  GIF: normal
│  设计意图: 挂件在"陪伴"状态，随时准备响应用户
│
├─ 30s ~ 2min ─ "轻度无聊"
│  行为: 微动幅度增大到 ±1.5px（pendant 开始"不安分"）
│        + 偶尔(概率 15%/30s)短暂切换到 look_around GIF（张望 2 秒）
│  GIF: normal 为主，偶尔 look_around
│  设计意图: 挂件开始注意到"没人理我"，稍微活动一下
│
├─ 2min ~ 10min ─ "中度等待"
│  行为: 微动恢复到 ±0.5px（安静下来）
│        + 每 30-60s 切换一次 stretch GIF（伸懒腰，#2）
│        + 绳索张力轻微降低（pendant 略微"放松"下垂，重力缩放从 0.6→0.55）
│  GIF: normal + stretch 交替
│  设计意图: 不焦虑也不兴奋，进入了"耐心等待"节奏
│
├─ 10min ~ 30min ─ "深度空闲"
│  行为: 进入打盹状态(#5)
│        微动减少到 ±0.2px（几乎不动）
│        重力缩放降到 0.5（pendant 更"重"更下垂）
│  GIF: sleep（闭眼/轻微呼吸缩放）
│  设计意图: 挂件已经"睡着"了，但保活响应——光标移动 >50px 立即唤醒
│
├─ >30min ────── "沉睡"
│  行为: 完全静止，微动关闭
│        + 每 2-5 分钟随机轻微抽动一下（sleep_twitch，单帧事件）
│  GIF: sleep
│  唤醒: 光标移动 >50px / 任何点击 → 1.5s 唤醒动画（sleep → yawn GIF → normal）
│  设计意图: 深度休眠。唤醒有仪式感——不突兀，但让用户注意到"它醒了"
```

### 唤醒过渡（防止骚扰感的关键）

```
沉睡中 → 检测到交互 → 不立即切 normal
       → 播放 yawn GIF（8 帧，1.5s，pendant 慢慢睁眼/伸展）
       → 过渡结束后切换到 normal
       → 重置无交互计时器
```

**设计原则**：
- 唤醒不是瞬时的——有"醒来"的过程，像真正的宠物
- 唤醒期间不响应额外交互（防止点击狂魔把 pendant 从深度睡眠直接打到 annoyed）
- 如果唤醒后 2 秒内无进一步交互，挂件进入"刚醒"状态：微动幅度比正常大 30%，持续 5 秒（"伸懒腰后的活跃期"）

### 需要的 GIF 清单

| GIF | 帧数 | 用途 |
|-----|------|------|
| look_around | ~15 | 轻度无聊张望 |
| stretch | ~15 | 中度等待伸懒腰 |
| sleep | ~20 | 深度空闲/沉睡 |
| yawn | ~8 | 唤醒过渡 |
| sleep_twitch | ~3 | 沉睡中偶尔抽动 |

总计约 61 帧，分布在 5 个 GIF 中。通过 IL 注入嵌入资源是可行的（已有 EXE 中 5 个 GIF 的先例）。

### CPU 占用控制

- 所有计时器使用帧计数器（已有的 `_frameCounter` 模式），不需要 `System.Timers.Timer` 或额外线程
- 随机间隔用 `(frameCounter * 7919) % interval` 伪随机，不调 `Random.Next()`
- 沉睡状态下跳过物理更新（已有 freeze 检测），CPU 占用接近于零
- GIF 切换只在状态转换时发生，不影响稳态渲染成本

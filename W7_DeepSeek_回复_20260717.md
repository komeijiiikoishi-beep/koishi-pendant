## Q1: Verlet 绳索的当前参数合理性

### 参数逐项分析

**阻尼 0.9985 @ ~100fps**

每帧保留 99.85% 的速度。半衰期计算：
```
v(t) = v₀ * damping^t
v(t)/v₀ = 0.5 → t = ln(0.5) / ln(0.9985) ≈ 462 帧 ≈ 4.6 秒
```

这意味着 pendant 被扰动后，约 4.6 秒速度减半，约 15 秒后只剩 10%。考虑到绳索总长只有 ~N 个段（估计 8-15 段），这个衰减速度**偏慢**。对于桌面挂件来说，用户甩光标后 pendant 应该 2-3 秒内稳定下来才算响应灵敏。建议降到 0.996（半衰期 ~1.7 秒）。

但实际上，0.9985 这么高的值可能有意图——它让绳索在微小的光标移动中保持"流动性"，pendant 看起来一直在动而不是僵住。如果降低阻尼，pendant 更快静止，就需要依赖微扰动来弥补活物感。这是**物理准确性 vs 活物感**的权衡。

**重力缩放 0.6**

标准重力在 100fps 下：`g * dt² = 9.8 * 0.01² = 0.00098 px/frame²`——几乎不可见。重力缩放到 0.6 意味着实际加速度 `0.6 * 9.8 = 5.88 m/s²`，相当于月球重力（1.62 m/s²）的 3.6 倍，地球的 60%。这个值偏主观——它让 pendant 有一定的"重量感"但不会太重。合理，但要注意：重力缩放和 dt² 都是常数，实际上 `gravity * dt²` 这个乘积才是 Verlet 中真正起作用的量。在 100fps 下这个积 = 0.000588 px/frame²，太小，说明重力在每帧的影响微乎其微——绳索行为主要由锚点移动 + 约束求解驱动。

**锚点速度传递 0.56**

这个参数控制光标速度有多少传递给绳索第一个段。值 0.56 意味着光标移动时，绳索只"跟"了 56% 的速度。在 Verlet 中通过 `ropeOldPos[0] = ropePos[0] - cursorVelocity * 0.56 * dt` 或等效方式实现。

**参数矛盾**：高阻尼（0.9985）+ 高锚点传递（0.56）的组合意味着：
- 锚点快速移动时，速度通过 0.56 系数注入绳段 0
- 然后每帧保留 99.85%——几乎不衰减
- 结果：pendant 被甩出去后摆动很久（因为高阻尼保留速度）但摆动幅度被锚点传递限制（因为每次只传递 56%）

这不是矛盾，而是**产生了特定手感**：pendant 有"惯性记忆"——它记得之前的运动，但新运动需要较大光标移动才能注入显著速度。表现为 pendant "有点迟钝但一旦动起来就不容易停"。

如果要更紧致的响应感：阻尼 0.995 + 锚点传递 0.70 → pendant 更紧贴光标，2 秒内稳定。

**约束迭代动态计算**

约束求解器每次迭代将相邻段拉回到 segmentLength。迭代次数基于"速度和段数"动态计算——这很合理，高速运动时增加迭代防止拉伸。但要注意：如果动态计算的上限不够高，在极端甩动时绳索会明显拉长（拉伸守卫 1.18x 触发）。建议确认动态公式在高速度下的行为。

**静止检测 0.05px + 1° + 0.8s**

三个条件同时满足才冻结。0.05px 的阈值非常严格（亚像素精度），1° 的角度偏差也很紧。0.8s 的持续时间（P4b 从 0.3s 改过来）在 100fps 下是 80 帧——合理，防止 pendant 在微风中反复冻结/解冻。

潜在问题：0.05px 阈值在 float32 下可能会因为浮点舍入误差永远达不到。如果 `ropeX` 的值在 ±2000 范围内（多屏幕），float32 的精度大约是 0.0002，所以 0.05 是可行的。但绳索末段经过多段约束求解后，相邻两帧的位置差可能会因为约束求解的迭代残差而略大于 0.05，导致静止检测永远不触发。建议：如果没有实际观察到"永远不冻结"的问题，就保持现状。如果有，把阈值放宽到 0.1px。

### 调整建议汇总

| 参数 | 当前值 | 建议 | 理由 |
|------|--------|------|------|
| 阻尼 | 0.9985 | 0.9965（如果 pendant 摆动太久）或保持（如果喜欢流动感） | 当前半衰期 4.6s |
| 锚点传递 | 0.56 | 0.62（如果感觉 pendant "跟不上"光标） | 当前较保守 |
| 重力缩放 | 0.6 | 保持 | 主观参数，没有明显问题 |
| 静止阈值 | 0.05px | 保持，除非观察到永不冻结 | float32 精度边界 |
| 拉伸守卫 | 1.18x / 1.32x | 保持 | 合理的两级保护 |

## Q2: 碰撞响应的物理正确方式

### 弹性碰撞的 Verlet 操作

从 build_r1.py 第 149-158 行的当前实现出发。先理解当前在做什么：

```
碰撞检测: ropeY[idx] + 20 > platformTop  (pendant 末点越过平台顶面)
当前响应: ropeY[idx] = platformTop - 20     ← 位置修正
          ropeOldY[idx] = platformTop - 20  ← 历史位置同值，隐式速度=0
```

**问题**：`ropeOldY[idx] = ropeY[idx]` 使得 `v_implicit = pos - oldPos = 0`。下一帧无论重力如何，都是从零速开始。pendant 直接粘在表面上。

### 正确的弹性碰撞实现

**Step 1: 保存碰撞前的状态**

```
penetratingY = ropeY[idx]           // 穿透后的 Y（碰撞前的位置）
vy = penetratingY - ropeOldY[idx]    // 隐式速度（正方向=向下）
                                     // 注意：ropeOldY[idx] 还是上一帧的值，未被修改
```

**Step 2: 位置修正**

```
ropeY[idx] = platformTop - pendantRadius  // 修正到接触面
```

**Step 3: 速度反射**

对于恢复系数 `r`（建议 0.3-0.5）：

```
ropeOldY[idx] = ropeY[idx] + vy * r
```

推导：
- 反射后目标速度：`vy' = -vy * r`（翻转方向 + 衰减）
- 需要设置的 oldPos 使 `newPos - oldPos = vy'`
- 所以：`oldPos = newPos - vy' = ropeY[idx] - (-vy * r) = ropeY[idx] + vy * r`

**数值示例**：pendant 以 15 px/frame 的速度落到平台顶面，r = 0.4：
```
vy = 15
ropeY[idx] = platformTop - 20     // 修正后的位置
ropeOldY[idx] = ropeY[idx] + 15 * 0.4 = ropeY[idx] + 6
```
结果：下一帧隐式速度 = `ropeY[idx] - (ropeY[idx] + 6) = -6 px/frame`（向上），反弹速度是原速度的 40%，符合预期。

**Step 4: 切向滑动**

```
// X 方向碰撞边界（pendant 不穿透窗口左右边缘）
if ropeX[idx] < platformLeft + pendantR:
    penetratingX = ropeX[idx]
    vx = penetratingX - ropeOldX[idx]
    ropeX[idx] = platformLeft + pendantR
    ropeOldX[idx] = ropeX[idx] - vx * r      // X 方向也反射

if ropeX[idx] > platformRight - pendantR:
    // 对称处理
```

对于**沿顶面滑动**（pendant 落在顶部但横向移动）：
- X 方向不受碰撞约束——自然滑动。
- 可选：施加 Coulomb 摩擦 —— vx 乘以摩擦系数（0.85-0.95）：

```
ropeOldX[idx] = ropeX[idx] - vx * friction
```

**IL 实现**：这些操作全部是 float 加减乘 + 条件跳转，可以在 IL 中直接实现。但参数 r（恢复系数）和 friction（摩擦系数）最好是可调的——建议通过 WinPlat.dll 的新方法 `BounceResponse(ref float pos, ref float oldPos, float surface, float radius, float restitution)` 封装，IL 层只调一个方法调用。

### 避免碰撞后抖动

抖动循环的机制：

```
帧 N:   pendent 碰撞 → 位置修正到表面，速度反射为 vy' = -vy * r
帧 N+1: 重力加速 → pendent 再次穿透 → 再次反射 → 速度再次翻转
帧 N+2: 同上，产生高频振荡
```

**三项对策**：

**对策 1: 最小速度阈值**

```python
if abs(vy * r) < v_min:  # v_min = 1.0 px/frame
    # 反弹太小，直接进入 resting contact
    ropeOldY[idx] = ropeY[idx]  # 零隐式速度
else:
    ropeOldY[idx] = ropeY[idx] + vy * r  # 正常反弹
```

`v_min = 1.0 px/frame` 意味着反弹速度低于 1px/frame 时放弃反弹。这对应于"弹不起来了就停"的物理直觉。在 100fps 下，1px/frame = 100px/s——这个阈值覆盖了肉眼可见的微弹。

**对策 2: 穿透容差**

```python
# 不要严格修正到 platformTop - pendantR
# 留 0.3px 微小间隙
ropeY[idx] = platformTop - pendantR + 0.3
```

这 0.3px 的间隙防止浮点舍入导致的假阳性碰撞。0.3px 在屏幕上小于一个像素，肉眼完全不可见。

**对策 3: Resting Contact 标记**

```csharp
bool _restingOnPlatform = false;

// 在碰撞响应中：
if (!_restingOnPlatform) {
    // 正常弹性碰撞
    vy = ...;
    if (abs(vy * r) < v_min) {
        _restingOnPlatform = true;  // 进入 resting
        ropeOldY[idx] = ropeY[idx];
    } else {
        // 正常反射
    }
} else {
    // Resting 模式：只修正穿透，不反弹
    ropeY[idx] = platformTop - pendantR + 0.3;
    ropeOldY[idx] = ropeY[idx];  // 保持零速
    
    // 检测是否应该解除 resting
    if (ropeY[idx] - ropeOldY[idx] > 0.1) {  // pendant 开始下落
        _restingOnPlatform = false;
    }
    // 或：锚点向上移动超过阈值 → 解除 resting
    if (cursorDeltaY < -5.0) {
        _restingOnPlatform = false;
    }
}
```

**抖动防护的 IL 实现**：对策 1 和 2 只需要额外加一两个条件跳转和 float 比较，IL 可行。对策 3 需要额外字段（类似 P4c 的 `_frozenMicroPhase`），也是 IL 可行的——但三者的组合逻辑较复杂，建议封装到 WinPlat.dll 的新方法中。

## Q3: 微扰动的替代方案

### 当前方案的评估

P4c 的锯齿波微扰动：

```
t = (_frozenMicroPhase++) % 300    // 300帧周期 → 约3秒
ox = (t - 150) * 0.0035            // ±0.525px, 线性锯齿波
oy = ((t + 75) % 300 - 150) * 0.0025  // ±0.375px, Y相位偏移75帧
```

**问题**：
1. **锯齿波是线性的**——pendant 匀速移过去、跳回来、再匀速移过去。这个"跳回"瞬间（t 从 299 到 300 → ox 从 +0.525 跳变到 -0.525）是**肉眼可感知的微小抽搐**。
2. **固定周期**——3秒的周期创造了可预测的节律。如果用户盯着看几秒，就能发现"它在按固定节奏晃"。真正活物的微动没有固定周期。
3. **XY 相关性**——X 和 Y 的相位差是固定的 75 帧（= 0.25 周期），创造了一个**固定的 Lissajous 轨道**（实际上是倾斜的椭圆 + 锯齿波跳回），看起来像 pendent 在画一个不变的图形。

### 替代方案对比

#### 方案 A: 双频正弦叠加（基于物理/数学，推荐）

```
ox = sin(t * 0.016) * 0.8 + sin(t * 0.089 + 1.7) * 0.3
oy = sin(t * 0.022 + 0.7) * 0.6 + sin(t * 0.103) * 0.2
```

- 主频率 ~0.25Hz（缓慢漂移，模拟呼吸）+ 副频率 ~1.4Hz（微颤，模拟肌肉微张力）
- 总幅度 ±1.1px X，±0.8px Y
- 双频叠加**不会产生周期性跳变**——sin 是平滑的
- 两个频率不可约简（1.6:8.9 不是简单整数比），所以轨道不会重复

**IL 实现**：
```
需要 Math.Sin() 调用 → 通过 Mono.Cecil 导入 mscorlib.System.Math.Sin
t 从帧计数器转换: float phase = _frozenMicroPhase * 0.016f
结果: sin(phase) * 0.8f + sin(phase * 5.56f + 1.7f) * 0.3f
```
- Mono.Cecil 可以 `mod.ImportReference(typeof(Math).GetMethod("Sin"))` 导入 Math.Sin
- 浮点运算（乘、加、Sin 调用）全部在 IL 层面可行
- Sin 调用频率：仅在冻结态每帧调用 4 次 Sin——约 400 次/秒，CPU 负担可忽略

**自然度**：★★★★★ — 不可预测、无跳变、无周期感

**侵入性**：低 — 只需替换 P4c 的扰动 IL 序列（约 65 条指令替换为约 80 条），Sin 调用更简洁

#### 方案 B: Perlin 噪声（理论上最优，但 IL 实现不现实）

Perlin 噪声的优势是空间/时间连续性 + 自然感。但实现 Perlin 噪声需要：
- 哈希函数（permutation table + bit manipulation）
- 梯度计算
- 平滑插值

在 IL 中实现这些 = 300+ 条 IL 指令，不可维护。

**替代**：可以用**三个不同频率的正弦波叠加**来近似 Perlin 噪声的效果：

```
ox = sin(t*0.016)*0.7 + sin(t*0.047+1.2)*0.4 + sin(t*0.121+2.8)*0.2
```

三个不可约简频率的叠加产生的运动轨道在人类观察尺度上（几秒到一两分钟）看起来是非周期性的。这是"穷人的 Perlin 噪声"。

**自然度**：★★★★☆ — 比双频好，但 IL 更复杂

**侵入性**：中 — 三频 Sin 调用约 100 条 IL，仍在可维护范围内

#### 方案 C: 纯视觉偏移（不改物理）

在渲染时给 pendant 的绘制位置加微小偏移，物理层完全不动。pendant 物理上冻结在某个位置，但渲染出来在微动。

**优势**：和物理完全解耦，实现最简单——在 WPF 的 DrawingVisual.Transform 上叠加一个 TranslateTransform。

**劣势**：
- 物理碰撞检测使用的是 ropeX/ropeY（未偏移值），pendant 的**视觉位置**和**物理位置**分离。如果 pendant 视觉上在平台上方但物理位置穿透了平台，碰撞响应会基于物理位置触发——可能导致"明明没碰到，pendant 却被弹开了"的违和感。
- 微动不参与绳索动力学——pendant 微动时绳索保持静止，看起来像"pendant 自己在晃但绳子不动"，破坏物理一致性。

**自然度**：★★☆☆☆ — 视觉和物理的分离会被注意到

**侵入性**：极低 — 只改渲染，不动物理

### 推荐

**方案 A（双频正弦叠加）**，理由：

1. 自然度最高（不可预测、无跳变、无周期感）
2. IL 可行——Math.Sin 可通过 mscorlib 导入，float 运算量不比当前锯齿波多多少
3. 保留了物理一致性——微动通过 ropeX/ropeY 驱动，绳索约束求解器会传播这些微动到整条绳索，pendant 和绳索一起微动，看起来像 pendant 带动了绳子
4. 参数可调——频率、幅度、相位都可以通过修改注入的 float 常量来调节

**参数调优建议**：
- 主呼吸频率 0.25-0.35Hz（接近人类的静息呼吸节奏）
- 微颤频率 1.2-1.8Hz（接近肌肉微张力震颤频率）
- X 轴幅度略大于 Y 轴（±1.0 vs ±0.7px），因为 pendent 水平方向的视觉容忍度更大
- Y 轴相位相对于 X 轴偏移 ~0.7 弧度，避免产生明显的椭圆轨道
